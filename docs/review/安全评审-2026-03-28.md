## 🔴 高危安全问题

### 1. ✅ Shell 命令注入漏洞 (CWE-78)

**严重程度**: 🔴 高危 (CVSS 8.1)

**问题描述**:
`lib/utils.py:make_git_env()` 函数使用 base64 + shell 脚本方式创建 GIT_ASKPASS 脚本。虽然使用了 base64 编码，但仍存在命令注入风险：

```python
# 原代码 (存在风险)
encoded = base64.b64encode(token.encode()).decode()
f.write(
    "#!/bin/sh\n"
    f"echo \"$(echo '{encoded}' | base64 -d)\"\n"  # shell 解释器可能被利用
)
```

**安全风险**:
- 如果 token 在 base64 编码前包含特殊字符（如反引号、$()等），可能导致命令注入
- shell 脚本解释器处理字符串时可能产生意外行为
- 攻击者如果能控制 token 值，可能执行任意命令

**修复方案**:
将 shell 脚本改为 Python 脚本，完全避免 shell 解释器：

```python
# 修复后代码
script_content = (
    "#!/usr/bin/env python3\n"
    "import sys\n"
    # Token 使用 repr() 转义，确保特殊字符安全
    f"sys.stdout.write({repr(token)})\n"
)
```

**修复位置**: `lib/utils.py:153-204`

**相关 CWE**: CWE-78 (OS Command Injection)

### 2. ✅ 临时文件权限竞态条件 (CWE-367)

**严重程度**: 🔴 高危 (CVSS 7.4)

**问题描述**:
原代码使用 `tempfile.mkstemp()` 创建临时文件后再用 `os.chmod()` 设置权限，存在 TOCTOU (Time-of-Check-Time-of-Use) 竞态条件：

```python
# 原代码 (存在竞态条件)
fd, askpass_path = tempfile.mkstemp(prefix="git_askpass_", suffix=".sh")
with os.fdopen(fd, "w") as f:
    f.write(script_content)
os.chmod(askpass_path, 0o700)  # 竞态窗口：文件已创建但权限未设置
```

**安全风险**:
- 在文件创建和权限设置之间，文件可能被其他用户读取
- 包含敏感 token 的临时文件可能泄露
- 在多用户共享系统上风险更高

**修复方案**:
使用 `os.open()` 原子性创建文件并设置权限：

```python
# 修复后代码
askpass_path = tempfile.mktemp(prefix="git_askpass_", suffix=".py")
# 原子性创建文件，设置仅当前用户可读写（0o600）
# 使用 O_EXCL 确保不会覆盖已存在的文件（防止 TOCTOU 攻击）
fd = os.open(askpass_path, os.O_CREAT | os.O_EXCL | os.O_WRONLY, 0o600)
try:
    os.write(fd, script_content.encode())
finally:
    os.close(fd)
# 设置可执行权限（保持只有所有者可读写执行）
os.chmod(askpass_path, 0o700)
```

**修复位置**: `lib/utils.py:181-199`

**相关 CWE**: CWE-367 (TOCTOU Race Condition), CWE-378 (Temporary File Issues)

### 3. ✅ 路径遍历漏洞 (CWE-22)

**严重程度**: 🔴 高危 (CVSS 7.5)

**问题描述**:
仓库名称直接从 API 获取后未经验证就用于构建文件路径和 git 命令，存在路径遍历风险：

```python
# 原代码 (缺少输入验证)
repo_name = repo["name"]  # 直接使用 API 返回的数据
temp_dir = tempfile.mkdtemp(prefix=f"sync_{repo_name}_")  # 可能包含 ../
```

**安全风险**:
- 恶意仓库名（如 `../../etc/passwd`）可能导致路径遍历
- 可能覆盖系统关键文件
- 虽然 API 通常返回合法名称，但不应完全信任外部输入

**修复方案**:
添加仓库名称验证函数，拒绝包含特殊字符的仓库名：

```python
def validate_repo_name(repo_name):
    """验证仓库名称的安全性，防止路径遍历和注入攻击"""
    if not repo_name or not isinstance(repo_name, str):
        return False
    if len(repo_name) < 1 or len(repo_name) > 100:
        return False
    if '..' in repo_name or '/' in repo_name or '\\' in repo_name:
        return False
    if repo_name.startswith('.') or repo_name.endswith('.'):
        return False
    # 仅允许字母、数字、连字符、下划线、点号
    import string
    allowed = string.ascii_letters + string.digits + '-_.'
    if not all(c in allowed for c in repo_name):
        return False
    return True
```

在同步流程中应用验证：

```python
# sync.py 中添加验证
if not validate_repo_name(repo_name):
    logging.warning(
        f"[{idx}/{total}] Skipping {repo_name}: "
        f"invalid repository name (security check)"
    )
    skipped += 1
    continue
```

**修复位置**: `lib/utils.py:120-159`, `sync.py:328-336`

**相关 CWE**: CWE-22 (Path Traversal), CWE-20 (Improper Input Validation)

### 4. ✅ Token 脱敏不完整 (CWE-532)

**严重程度**: 🔴 高危 (CVSS 7.2)

**问题描述**:
原有的 `mask_token()` 函数仅处理 URL 中的 token（`https://token@host`），未覆盖其他 token 暴露场景：

```python
# 原代码 (脱敏不完整)
def mask_token(text):
    return re.sub(r'https://[^@]+@', 'https://***@', str(text))
```

**安全风险**:
- 查询参数中的 token（`?access_token=xxx`）未被脱敏
- GitHub PAT 格式（`ghp_*`, `gho_*`, `github_pat_*`）未被检测
- Bearer token header 中的 token 未被脱敏
- 异常堆栈、第三方库日志可能泄露 token

**修复方案**:
增强 `mask_token()` 函数，覆盖所有常见 token 格式：

```python
def mask_token(text):
    """增强的 token 脱敏，覆盖多种格式"""
    if not text:
        return text
    text = str(text)

    # 1. URL 中内嵌的凭据
    text = re.sub(r'https://[^@\s]+@', 'https://***@', text)

    # 2. 查询参数中的 token
    text = re.sub(r'[?&]access_token=[^&\s]+', '?access_token=***', text)

    # 3. GitHub Personal Access Tokens
    text = re.sub(r'ghp_[a-zA-Z0-9]{36}', 'ghp_***', text)
    text = re.sub(r'gho_[a-zA-Z0-9]{36}', 'gho_***', text)
    text = re.sub(r'github_pat_[a-zA-Z0-9_]{82}', 'github_pat_***', text)

    # 4. Bearer tokens in headers
    text = re.sub(r'Bearer\s+[a-zA-Z0-9_\-\.]{20,}', 'Bearer ***', text)

    # 5. 通用长字符串 token (长度>40)
    text = re.sub(
        r'\b[a-zA-Z0-9_\-]{40,}\b',
        lambda m: m.group(0)[:8] + '***' if len(m.group(0)) > 40 else m.group(0),
        text
    )

    return text
```

**修复位置**: `lib/utils.py:167-212`

**相关 CWE**: CWE-532 (Information Exposure Through Log Files), CWE-200 (Information Exposure)

**补充**: `TokenMaskingFilter` 已在二级评审中添加，作为日志脱敏的最后一道防线。

## 🟡 中危安全问题

### 5. ✅ 敏感数据残留风险 (CWE-459)

**严重程度**: 🟡 中危 (CVSS 5.9)

**问题描述**:
临时 askpass 脚本删除时未覆盖文件内容，在文件系统级别可能被恢复：

```python
# 原代码 (直接删除，数据可恢复)
try:
    os.unlink(p)
except OSError:
    pass
```

**安全风险**:
- 删除的文件在文件系统层面可能被恢复（尤其是 SSD 和现代文件系统）
- 包含敏感 token 的脚本内容可能被攻击者恢复
- 在共享主机或云环境中风险较高

**修复方案**:
先用零覆盖文件内容，再删除文件：

```python
for p in askpass_paths:
    try:
        if os.path.exists(p):
            # 用零覆盖文件内容
            file_size = os.path.getsize(p)
            with open(p, 'wb') as f:
                f.write(b'\x00' * file_size)
            os.unlink(p)
    except OSError as e:
        logging.warning(f"  Failed to clean askpass script {p}: {e}")
```

**修复位置**: `lib/sync_repo.py:209-219`

**相关 CWE**: CWE-459 (Incomplete Cleanup), CWE-226 (Sensitive Information Uncleared Before Release)

### 6. ✅ 临时目录清理失败静默忽略

**严重程度**: 🟡 中危 (CVSS 4.3)

**问题描述**:
临时目录清理使用 `ignore_errors=True`，清理失败时不记录任何日志：

```python
# 原代码 (静默忽略清理失败)
shutil.rmtree(temp_dir, ignore_errors=True)
```

**安全风险**:
- 清理失败导致包含仓库代码的临时目录残留
- 私有仓库代码可能泄露到共享文件系统
- 磁盘空间可能被耗尽
- 运维人员无法发现清理失败问题

**修复方案**:
记录清理失败的警告，便于诊断：

```python
try:
    shutil.rmtree(temp_dir, ignore_errors=False)
except Exception as e:
    logging.warning(
        f"  Failed to clean up temp directory {temp_dir}: {e}"
    )
```

**修复位置**: `lib/sync_repo.py:201-207`

**相关 CWE**: CWE-459 (Incomplete Cleanup)

## 附录: 安全编码最佳实践

本次修复中应用的安全编码原则：

1. **最小权限原则**: 临时文件权限设置为 0o700（仅所有者）
2. **纵深防御**: 多层 token 脱敏（mask_token + TokenMaskingFilter）
3. **输入验证**: 所有外部输入（仓库名）都经过验证
4. **安全默认值**: 默认行为即为安全行为（失败时记录警告而非静默）
5. **最小化攻击面**: 使用 Python 脚本代替 shell 脚本
6. **完整清理**: 敏感数据使用后立即清理（覆盖+删除）

**建议后续开发者遵循这些原则**。
