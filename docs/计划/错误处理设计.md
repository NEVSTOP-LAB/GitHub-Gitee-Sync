# 错误处理设计

## 概述

本文档对 GitHub-Gitee 同步过程中可能出现的各种错误场景进行预分析，并给出对应的解决方案和处理策略。

---

## 1. 错误分类

将可能出现的错误分为以下 5 大类：

| 类别 | 严重级别 | 影响范围 | 处理策略 |
|------|---------|---------|---------|
| **认证错误** | 🔴 致命 | 全局 | 立即退出 |
| **API 错误** | 🟡 严重 | 单个仓库或全局 | 重试或跳过 |
| **Git 操作错误** | 🟡 严重 | 单个仓库 | 跳过当前仓库 |
| **网络错误** | 🟠 可恢复 | 单个请求 | 重试 |
| **数据错误** | 🟢 轻微 | 单个仓库的附属信息 | 记录日志继续 |

---

## 2. 认证错误

### 2.1 Token 无效或过期

**场景**：
- GitHub/Gitee Personal Access Token 已过期
- Token 被用户手动撤销
- Token 格式错误

**检测方式**：
```python
# 在同步开始前进行 Token 验证
# GitHub: GET /user
resp = requests.get("https://api.github.com/user", headers={"Authorization": f"token {token}"})
if resp.status_code == 401:
    raise AuthError("GitHub Token 无效或已过期")

# Gitee: GET /api/v5/user
resp = requests.get(f"https://gitee.com/api/v5/user?access_token={token}")
if resp.status_code == 401:
    raise AuthError("Gitee Token 无效或已过期")
```

**处理方案**：
- ❌ 不重试（Token 无效重试无意义）
- ❌ 不继续（所有后续操作都需要 Token）
- ✅ 立即退出，输出清晰的错误信息，指导用户检查 Token

**用户提示**：
```
[FATAL] GitHub Token 认证失败 (HTTP 401)
        请检查 Token 是否有效: https://github.com/settings/tokens
        所需权限: repo (完整仓库访问)
```

### 2.2 Token 权限不足

**场景**：
- Token 缺少 `repo` scope（无法访问私有仓库）
- Token 缺少 `read:org` scope（无法列出组织仓库）
- Gitee Token 缺少 `projects` 权限

**检测方式**：
- GitHub: 检查响应头 `X-OAuth-Scopes`
- Gitee: 检查响应状态码 403

**处理方案**：
- 如果 `include_private=true` 但 Token 无 `repo` scope → 降级为仅同步公开仓库，输出警告
- 如果 `account_type=org` 但缺少 `read:org` scope → 报错退出

## 3. API 错误

### 3.1 Rate Limit（速率限制）

**场景**：
- GitHub API 5000 次/小时限额用尽
- Gitee API 返回 429 Too Many Requests

**检测方式**：
```python
# GitHub: 检查响应头
remaining = int(resp.headers.get("X-RateLimit-Remaining", 999))
reset_time = int(resp.headers.get("X-RateLimit-Reset", 0))

if resp.status_code == 403 and remaining == 0:
    wait_seconds = reset_time - time.time()
    # Rate limit exceeded
```

**处理方案**：

| 剩余配额 | 策略 |
|---------|------|
| > 100 | 正常执行 |
| 10 ~ 100 | 每个请求间增加 1 秒延迟 |
| 0 | 等待重置（最多等待 15 分钟），超时则退出 |

```python
def handle_rate_limit(resp):
    remaining = int(resp.headers.get("X-RateLimit-Remaining", 999))
    if remaining < 100:
        time.sleep(1)  # 加入延迟
    if remaining == 0:
        reset_time = int(resp.headers.get("X-RateLimit-Reset", 0))
        wait = max(0, reset_time - time.time())
        if wait > 900:  # 超过 15 分钟
            raise RateLimitError("API 配额已用尽，重置时间过长")
        logging.warning(f"API 配额已用尽，等待 {wait:.0f} 秒...")
        time.sleep(wait + 1)
```

### 3.2 仓库创建失败

**场景**：
- 目标平台已存在同名仓库（API 列表未获取到，但实际存在）
- 仓库名包含目标平台不支持的字符
- 组织账号下创建仓库权限不足

**处理方案**：

| 状态码 | 原因 | 处理 |
|-------|------|------|
| 422 | 仓库已存在 | 忽略错误，继续同步（视为仓库已存在） |
| 403 | 权限不足 | 记录错误，跳过该仓库 |
| 400 | 参数无效（仓库名不合法） | 记录错误，跳过该仓库 |

```python
def create_repo_safe(owner, token, repo_name, ...):
    resp = create_gitee_repo(...)
    if resp.status_code == 422:
        # 仓库可能已存在，不视为错误
        logging.info(f"仓库 {repo_name} 在目标平台已存在，跳过创建")
        return True
    if resp.status_code >= 400:
        logging.error(f"创建仓库 {repo_name} 失败: {resp.status_code} {resp.text}")
        return False
    return True
```

### 3.3 获取仓库列表失败

**场景**：
- API 返回非 200 状态码
- 返回数据格式异常
- Owner 不存在

**处理方案**：
- 重试 3 次（间隔递增：2s, 4s, 8s）
- 3 次都失败 → 退出程序（无法继续）

### 3.4 API 返回数据异常

**场景**：
- 返回的 JSON 结构不符合预期
- 某些字段缺失

**处理方案**：
- 使用 `.get()` 安全访问字段，提供默认值
- 关键字段缺失时跳过该仓库

```python
repo_name = repo.get("name")
if not repo_name:
    logging.warning(f"跳过无效仓库数据: {repo}")
    continue
```

## 4. Git 操作错误

### 4.1 git clone --mirror 失败

**可能原因及对应方案**：

| 原因 | 错误信息（关键词） | 处理 |
|------|-----------------|------|
| Token 认证失败 | `Authentication failed` | 已在启动时验证，此处跳过该仓库 |
| 仓库不存在 | `Repository not found` | 跳过该仓库，记录警告 |
| 空仓库 | `warning: You appear to have cloned an empty repository` | 特殊处理：标记为空仓库，跳过 push |
| 网络超时 | `Could not resolve host` / `Connection timed out` | 重试 2 次 |
| 磁盘空间不足 | `No space left on device` | 记录错误，尝试清理临时目录 |
| 仓库过大 | 超时 | 增大超时时间，或跳过 |

```python
def mirror_clone(source_url, local_path, timeout=600):
    try:
        result = subprocess.run(
            ["git", "clone", "--mirror", source_url, local_path],
            capture_output=True, text=True, timeout=timeout
        )
        if result.returncode != 0:
            stderr = result.stderr
            if "empty repository" in stderr.lower():
                return "empty"
            raise GitError(f"clone 失败: {mask_token(stderr)}")
        return "success"
    except subprocess.TimeoutExpired:
        raise GitError(f"clone 超时 ({timeout}s)")
```

### 4.2 git push --mirror 失败

**可能原因及对应方案**：

| 原因 | 错误信息（关键词） | 处理 |
|------|-----------------|------|
| Token 认证失败 | `Authentication failed` | 跳过该仓库 |
| 目标仓库不存在 | `Repository not found` | 如果 `create_missing_repos` 已开启但创建失败，跳过 |
| 远端拒绝推送 | `remote rejected` | 记录详细错误信息，跳过 |
| 受保护分支 | `protected branch hook declined` | 记录警告，Gitee 默认分支保护可能导致此问题 |
| 推送包过大 | `pack exceeds maximum allowed size` | 记录错误，跳过（需用户手动处理） |
| 网络中断 | `Connection reset` / `Broken pipe` | 重试 2 次 |

### 4.3 Token 泄露防护

在日志中打印 Git 命令输出时，必须屏蔽 Token：

```python
def mask_token(text):
    """将日志中的 Token 替换为 ***"""
    import re
    # 匹配 URL 中的 token
    text = re.sub(r'https://[^@]+@', 'https://***@', text)
    return text
```

## 5. 网络错误

### 5.1 通用网络错误处理

| 错误类型 | 重试次数 | 间隔策略 | 超时设置 |
|---------|---------|---------|---------|
| `ConnectionError` | 3 次 | 指数退避 (2s, 4s, 8s) | - |
| `Timeout` | 3 次 | 指数退避 (2s, 4s, 8s) | API: 30s, Git: 600s |
| `ConnectionResetError` | 3 次 | 固定 5s | - |
| DNS 解析失败 | 1 次 | 10s | - |

### 5.2 重试装饰器设计

```python
import time
import requests

def retry(max_retries=3, backoff_base=2, exceptions=(requests.RequestException,)):
    def decorator(func):
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    last_error = e
                    if attempt < max_retries:
                        wait = backoff_base ** attempt
                        logging.warning(
                            f"{func.__name__} 失败 (第 {attempt+1} 次), "
                            f"{wait}s 后重试: {e}"
                        )
                        time.sleep(wait)
            raise last_error
        return wrapper
    return decorator

@retry(max_retries=3, backoff_base=2)
def api_get(url, **kwargs):
    resp = requests.get(url, timeout=30, **kwargs)
    resp.raise_for_status()
    return resp
```

## 6. 数据错误

### 6.1 仓库名兼容性

**场景**：GitHub 和 Gitee 对仓库名的限制不完全一致

| 平台 | 仓库名限制 |
|------|----------|
| GitHub | 字母、数字、`-`、`.`、`_`，最长 100 |
| Gitee | 字母、数字、`-`、`_`，某些特殊字符可能不支持 |

**处理方案**：
- 检测仓库名是否包含目标平台不支持的字符
- 如不支持，记录警告并跳过该仓库

### 6.2 Release Assets 同步错误

**场景**：
- 源平台附件过大，下载超时
- 目标平台有文件大小限制
- 下载/上传过程中网络中断

**处理方案**：
- 单个 asset 失败不影响其他 assets 的同步
- 记录失败的 asset 名称和原因

### 6.3 空仓库处理

**场景**：源仓库是空仓库（无任何 commit）

**处理方案**：
- `git clone --mirror` 会成功但创建的是空的 bare repo
- `git push --mirror` 到空目标仓库也能成功
- 如果目标仓库有 auto_init 内容，会被覆盖（这就是为什么创建时不要 auto_init）
- 空仓库的附属信息（releases 等）通常也为空，无需特殊处理

## 7. 双向同步冲突

### 7.1 同一仓库双向都有修改

**场景**：`github↔gitee` 双向同步时，两边都有新提交

**问题**：`git push --mirror` 是破坏性操作，后推送的一方会覆盖另一方

**处理方案**：
- **方案 A（推荐）：以一方为主**
  - 默认以 GitHub 为主（先 GitHub→Gitee，后 Gitee→GitHub 仅同步 GitHub 没有的仓库）
  - 通过 `--primary` 参数指定主方
- **方案 B：跳过有冲突的仓库**
  - 比较两边的 HEAD commit，如果不同则跳过并报警告
- **方案 C：人工介入**
  - 检测到双向都有变更时输出警告列表，不自动同步

**建议**：

双向模式下默认采用 **方案 A**：以源（`--primary` 指定）为主，另一方仅补充新仓库。如检测到代码冲突（两边 HEAD 不同），输出警告但仍以主方为准。

### 7.2 Release/Issue 编号冲突

**场景**：双向同步时两端都有不同的 releases/issues

**处理方案**：
- 以 `tag_name`（releases）或标题+标识（issues）作为匹配键
- 匹配到的做更新，未匹配到的做创建
- 双向都创建的同名 release 以主方内容为准

## 8. 运行环境错误

### 8.1 Git 未安装

**检测**：
```python
try:
    subprocess.run(["git", "--version"], capture_output=True, check=True)
except FileNotFoundError:
    raise EnvironmentError("Git 未安装或不在 PATH 中")
```

### 8.2 磁盘空间不足

**检测**：
```python
import shutil
free_space = shutil.disk_usage("/tmp").free
if free_space < 1024 * 1024 * 100:  # 小于 100MB
    logging.warning("磁盘剩余空间不足 100MB，大仓库同步可能失败")
```

### 8.3 临时目录清理

**策略**：
- 使用 `tempfile.mkdtemp()` 创建临时目录
- 在 `finally` 块中确保清理
- 程序异常退出时可能遗留临时目录，下次运行前清理

```python
import tempfile
import shutil

temp_dir = tempfile.mkdtemp(prefix="sync_")
try:
    mirror_clone(source_url, temp_dir)
    mirror_push(target_url, temp_dir)
finally:
    shutil.rmtree(temp_dir, ignore_errors=True)
```

## 9. 错误报告与日志

### 9.1 日志级别

| 级别 | 使用场景 |
|------|---------|
| `FATAL` | Token 无效、环境缺失等致命错误，立即退出 |
| `ERROR` | 单个仓库同步失败、API 请求最终失败 |
| `WARNING` | 非致命问题（空仓库、附属信息同步失败、Rate Limit 临近） |
| `INFO` | 正常进度信息（开始同步、完成同步、统计数据） |
| `DEBUG` | 详细调试信息（API 请求/响应、Git 命令输出） |

### 9.2 同步结果报告

```
===== 同步结果 =====
方向: GitHub → Gitee
总计: 25 个仓库
  ✅ 成功: 22
  ❌ 失败: 2
  ⏭️  跳过: 1

失败仓库:
  - repo-a: git push --mirror 失败 (remote rejected)
  - repo-b: Gitee 创建仓库失败 (422 仓库名已存在)

跳过仓库:
  - repo-c: 目标仓库不存在且 create_missing_repos=false

附属信息同步警告:
  - repo-d: Wiki 同步失败 (源仓库无 Wiki)
  - repo-e: Release "v2.0" 附件 "dist.zip" 上传失败 (文件过大)
====================
```

### 9.3 退出码设计

| 退出码 | 含义 |
|-------|------|
| 0 | 全部成功 |
| 1 | 部分仓库失败（有成功的） |
| 2 | 全部失败 |
| 3 | 致命错误（认证失败、环境异常） |

| Token 无效 | 🔴 | ❌ | ❌ | ✅ | FATAL |
| Token 权限不足 | 🔴 | ❌ | ❌ | ✅ | FATAL |
| Git 未安装 | 🔴 | ❌ | ❌ | ✅ | FATAL |
| 获取仓库列表失败 | 🔴 | ✅(3次) | ❌ | ✅ | ERROR |
| Rate Limit | 🟠 | ✅(等待) | ❌ | 超时则退出 | WARNING |
| 网络超时 | 🟠 | ✅(3次) | ❌ | ❌ | WARNING |

| git push 失败 | 🟡 | ✅(2次) | ✅ | ❌ | ERROR |
| 空仓库 | 🟢 | ❌ | ✅ | ❌ | WARNING |
