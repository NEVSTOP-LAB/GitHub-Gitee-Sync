# 开发步骤

## 概述

本文档列出了实现 GitHub-Gitee 全量仓库同步工具的详细开发步骤。

---

## 阶段一：Python 同步脚本开发

### Step 1：项目初始化

- [ ] 创建 `requirements.txt`，添加 `requests` 依赖
- [ ] 创建 `sync.py` 主脚本文件
- [ ] 创建 `.gitignore`（排除 `.env`、`__pycache__` 等）

### Step 2：参数解析模块

- [ ] 实现 `parse_args()` 函数
  - 支持 CLI 参数：`--github-owner`, `--github-token`, `--gitee-owner`, `--gitee-token`, `--account-type`, `--include-private`, `--exclude-repos`
  - 支持环境变量：`GITHUB_OWNER`, `GITHUB_TOKEN`, `GITEE_OWNER`, `GITEE_TOKEN`, `ACCOUNT_TYPE`, `INCLUDE_PRIVATE`, `EXCLUDE_REPOS`
  - CLI 参数优先于环境变量

### Step 3：GitHub API 模块

- [ ] 实现 `get_github_repos()` 函数
  - 根据 `account_type` 选择 API 端点（`/user/repos` 或 `/orgs/{org}/repos`）
  - 分页获取全部仓库
  - 根据 `include_private` 过滤仓库
  - 返回仓库列表（name, private, description, clone_url）

### Step 4：Gitee API 模块

- [ ] 实现 `get_gitee_repos()` 函数
  - 根据 `account_type` 选择 API 端点
  - 分页获取全部仓库
  - 返回仓库名集合（用于快速查找）

- [ ] 实现 `create_gitee_repo()` 函数
  - 根据 `account_type` 选择创建端点（`/user/repos` 或 `/orgs/{org}/repos`）
  - 设置仓库名、描述、私有属性
  - **不要** 设置 `auto_init`（避免与 mirror push 冲突）

### Step 5：Git Mirror 同步模块

- [ ] 实现 `mirror_sync()` 函数
  - 使用 `subprocess` 执行 `git clone --mirror`
  - 使用 `subprocess` 执行 `git push --mirror`
  - 使用 `tempfile.mkdtemp()` 创建临时目录
  - 同步完成后清理临时目录
  - 在 URL 中嵌入 Token 进行认证
  - 日志中隐藏 Token 信息

### Step 6：主同步流程

- [ ] 实现 `sync_all()` 函数
  - 调用上述模块，串联完整同步流程
  - 输出进度信息（[1/N] 同步 repo-name ...）
  - 统计成功/失败/跳过数量
  - 错误不中断，记录后继续下一个仓库

### Step 7：错误处理与日志

- [ ] 配置 `logging` 模块
- [ ] API 请求添加重试逻辑（3 次，间隔递增）
- [ ] git 命令超时处理
- [ ] 友好的错误信息输出

## 关键设计决策

| 决策项 | 选择 | 原因 |
|-------|------|------|
| HTTP 库 | `requests` | 简洁易用，社区广泛 |
| Git 操作 | `subprocess` | 无额外依赖，可靠稳定 |
| 同步方式 | `git clone --mirror` + `git push --mirror` | 完整同步所有分支和标签 |
| 认证方式 | HTTPS + Token in URL | Docker 场景下最简单 |
| Docker 基础镜像 | `python:3.11-slim` | 体积适中，兼容性好 |
| 参数传递 | 环境变量 + CLI 参数 | 灵活，兼顾安全 |
| 错误处理 | 跳过失败仓库，继续同步 | 不因单个仓库失败影响全局 |
