# 开发步骤

## 概述

本文档列出了实现 GitHub-Gitee 全量仓库同步工具的详细开发步骤。

---

## 阶段一：Python 同步脚本开发

### Step 1：项目初始化

- [ ] 创建 `requirements.txt`，添加 `requests` 依赖
- [ ] 创建 `sync.py` 主脚本文件
- [ ] 创建 `.gitignore`（排除 `.env`、`__pycache__` 等）

### Step 2：参数解析模块

- [ ] 实现 `parse_args()` 函数
  - 支持 CLI 参数：`--github-owner`, `--github-token`, `--gitee-owner`, `--gitee-token`, `--account-type`, `--include-private`, `--exclude-repos`, `--direction`, `--create-missing-repos`, `--sync-extra`
  - 支持环境变量：`GITHUB_OWNER`, `GITHUB_TOKEN`, `GITEE_OWNER`, `GITEE_TOKEN`, `ACCOUNT_TYPE`, `INCLUDE_PRIVATE`, `EXCLUDE_REPOS`, `SYNC_DIRECTION`, `CREATE_MISSING_REPOS`, `SYNC_EXTRA`
  - CLI 参数优先于环境变量
  - `--direction` 支持 `github2gitee`、`gitee2github`、`both` 三种值

### Step 3：GitHub API 模块

- [ ] 实现 `get_github_repos()` 函数
  - 根据 `account_type` 选择 API 端点（`/user/repos` 或 `/orgs/{org}/repos`）
  - 分页获取全部仓库
  - 根据 `include_private` 过滤仓库
  - 返回仓库列表（name, private, description, clone_url）

### Step 4：Gitee API 模块

- [ ] 实现 `get_gitee_repos()` 函数
  - 根据 `account_type` 选择 API 端点
  - 分页获取全部仓库
  - 返回仓库名集合（用于快速查找）

- [ ] 实现 `create_gitee_repo()` 函数
  - 根据 `account_type` 选择创建端点（`/user/repos` 或 `/orgs/{org}/repos`）
  - 设置仓库名、描述、私有属性
  - **不要** 设置 `auto_init`（避免与 mirror push 冲突）
  - 仅在 `create_missing_repos=true` 时调用

### Step 4b：GitHub 仓库创建模块（反向/双向同步时需要）

- [ ] 实现 `create_github_repo()` 函数
  - 根据 `account_type` 选择创建端点（`/user/repos` 或 `/orgs/{org}/repos`）
  - 设置仓库名、描述、私有属性
  - **不要** 设置 `auto_init`（避免与 mirror push 冲突）
  - 仅在反向同步 (`gitee→github`) 或双向同步时使用

### Step 5：Git Mirror 同步模块

- [ ] 实现 `mirror_sync()` 函数
  - 使用 `subprocess` 执行 `git clone --mirror`
  - 使用 `subprocess` 执行 `git push --mirror`
  - 使用 `tempfile.mkdtemp()` 创建临时目录
  - 同步完成后清理临时目录
  - 在 URL 中嵌入 Token 进行认证
  - 日志中隐藏 Token 信息

### Step 6：主同步流程

- [ ] 实现 `sync_all()` 函数
  - 根据 `direction` 参数确定源/目标平台
  - 调用上述模块，串联完整同步流程
  - 输出进度信息（[1/N] 同步 repo-name ...）
  - 统计成功/失败/跳过数量
  - 错误不中断，记录后继续下一个仓库
  - 如果 `create_missing_repos=false`，跳过目标不存在的仓库
  - 如果是双向同步 (`both`)，正向完成后执行反向同步

### Step 7：错误处理与日志

- [ ] 配置 `logging` 模块
- [ ] 实现 Token 验证函数 `validate_tokens()`
- [ ] API 请求添加重试逻辑（3 次，间隔递增）
- [ ] git 命令超时处理（默认 600s）
- [ ] Rate Limit 检测与等待
- [ ] Token 信息日志脱敏 `mask_token()`
- [ ] 友好的错误信息输出
- [ ] 退出码设计（0=全部成功, 1=部分失败, 2=全部失败, 3=致命错误）

> 详细设计请参见 [错误处理设计](错误处理设计.md)

### Step 7b：仓库元信息同步模块

- [ ] 实现 `sync_repo_metadata()` 函数
  - 获取源仓库详情（description, homepage 等）
  - 对比目标仓库信息
  - 有差异时调用 PATCH API 更新

### Step 7c：附属信息同步模块

按优先级逐步实现，每个模块独立开发和测试。

#### Releases 同步（P1）

- [ ] 实现 `sync_releases()` 函数
  - 获取源/目标平台 releases 列表
  - 以 `tag_name` 为匹配键
  - 创建/更新目标端 releases
- [ ] 实现 `sync_release_assets()` 函数
  - 下载源平台 release assets
  - 上传到目标平台
  - 按文件名匹配，跳过已存在的同名 assets

#### Wiki 同步（P2）

- [ ] 实现 `sync_wiki()` 函数
  - 使用 `git clone --mirror {repo}.wiki.git` + `git push --mirror`
  - 源 wiki 不存在时静默跳过
  - 目标平台需已启用 wiki

#### Labels 同步（P2）

- [ ] 实现 `sync_labels()` 函数
  - 获取源/目标平台 labels 列表
  - 以 `name` 为匹配键
  - 同步 name、color、description

#### Milestones 同步（P2）

- [ ] 实现 `sync_milestones()` 函数
  - 获取源/目标平台 milestones 列表
  - 以 `title` 为匹配键
  - 同步 title、state、description、due_on

#### Issues 同步（P3，可选）

- [ ] 实现 `sync_issues()` 函数
  - 仅同步 open 状态的 issues
  - 在 body 中附加源链接标识，避免重复创建
  - 同步 issue comments
  - 记录编号映射关系

> 详细调研请参见 [仓库附属信息同步调研](../调研/仓库附属信息同步调研.md)

## 阶段三：GitHub Action 封装

### Step 10：创建 action.yml

- [ ] 创建 `action.yml` 元数据文件
  - 定义 inputs：`github-owner`, `github-token`, `gitee-owner`, `gitee-token`, `account-type`, `include-private`, `exclude-repos`, `direction`, `create-missing-repos`, `sync-extra`
  - 定义 outputs：`synced-count`, `failed-count`, `skipped-count`
  - 设置 `runs.using: 'docker'`，`runs.image: 'Dockerfile'`
  - 配置 branding（icon: `refresh-cw`, color: `blue`）

### Step 11：创建 entrypoint.sh

- [ ] 创建 `entrypoint.sh` 桥接脚本
  - 将 GitHub Action 的 `INPUT_*` 环境变量映射为 `sync.py` 期望的标准环境变量
  - 兼容非 Action 场景（直接使用标准环境变量时不受影响）
  - 调用 `python /app/sync.py`
  - 设置可执行权限 `chmod +x`

### Step 12：适配 Dockerfile

- [ ] 修改 Dockerfile 的 ENTRYPOINT 为 `entrypoint.sh`
- [ ] 确保 `entrypoint.sh` 被正确复制到镜像中
- [ ] 验证 Docker 独立运行和 GitHub Action 两种模式都正常

### Step 13：适配 sync.py 输出

- [ ] 在 `sync_all()` 函数中添加 `write_action_outputs()` 调用
  - 检测 `GITHUB_OUTPUT` 环境变量是否存在
  - 如存在，将同步结果写入 `$GITHUB_OUTPUT` 文件
  - 输出 `synced-count`、`failed-count`、`skipped-count`

## 关键设计决策

| 决策项 | 选择 | 原因 |
|-------|------|------|
| HTTP 库 | `requests` | 简洁易用，社区广泛 |
| Git 操作 | `subprocess` | 无额外依赖，可靠稳定 |
| 同步方式 | `git clone --mirror` + `git push --mirror` | 完整同步所有分支和标签 |
| 认证方式 | HTTPS + Token in URL | Docker 场景下最简单 |
| Docker 基础镜像 | `python:3.11-slim` | 体积适中，兼容性好 |
| 参数传递 | 环境变量 + CLI 参数 | 灵活，兼顾安全 |
| 错误处理 | 跳过失败仓库，继续同步 | 不因单个仓库失败影响全局 |
| Action 类型 | Docker Container Action | 复用已有 Dockerfile，完整环境控制 |
| Action 入口 | `entrypoint.sh` 桥接脚本 | 映射 `INPUT_*` 变量，兼容多种使用模式 |
| 同步方向 | 支持单向和双向 | `github2gitee`（默认）/ `gitee2github` / `both` |
| 仓库创建 | 可配置 `create_missing_repos` | 默认自动创建，可关闭仅同步已有仓库 |
| 附属信息 | 可选模块，按需启用 | 通过 `sync_extra` 参数控制，避免不需要时的额外开销 |
| Wiki 同步 | Git mirror 方式 | 与代码同步一致，简洁可靠 |
| Release 同步 | REST API | 两端 API 兼容，按 tag_name 匹配 |
| 双向冲突 | 以主方为准 | 默认 GitHub 为主，避免数据丢失 |
