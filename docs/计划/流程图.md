# 流程图

## 概述

本文档包含两个流程图及其详细说明：
1. **整体同步流程图** — 描述从启动到完成的全局流程
2. **单仓库同步细节流程图** — 描述一个仓库的完整同步过程（包括代码和附属信息）

### 整体流程说明

| 步骤 | 说明 | 关键逻辑 |
|------|------|---------|
| **1. 解析参数** | 按优先级读取：CLI 参数 > 环境变量 > 默认值 | `parse_args()` |
| **2. 验证参数** | 验证 Token 有效性、必填参数完整性 | 无效 Token 立即退出，不浪费时间 |
| **3. 确定同步方向** | 根据 `direction` 参数决定源/目标平台 | `github→gitee`、`gitee→github`、`github↔gitee` |
| **4. 获取源平台仓库列表** | 分页获取，一次最多 100 个，循环到获取完全部 | 根据 `account_type` 选择 API 端点 |
| **5. 过滤仓库** | 剔除用户指定不同步的仓库 | `exclude_repos` 逗号分隔的仓库名列表 |
| **6. 获取目标平台仓库列表** | 用于后续判断是否需要创建仓库 | 返回集合形式方便 O(1) 查找 |
| **7. 双向合并** | 仅在双向同步时需要，将两个列表合并去重 | 以仓库名为 key 合并 |
| **8. 遍历同步** | 逐个仓库调用同步逻辑，失败不中断 | 捕获异常，记录结果，继续下一个 |
| **9. 双向反向** | 双向模式下执行反方向同步 | 避免重复同步已处理的仓库 |
| **10. 输出摘要** | 打印统计数据和失败列表 | 帮助用户快速定位问题 |
| **11. Action Outputs** | 将结果写入 `$GITHUB_OUTPUT` | 仅在 GitHub Action 环境中生效 |

### 单仓库同步流程说明

| 步骤 | 说明 | 出错处理 |
|------|------|---------|
| **A. 检查/创建仓库** | 在目标平台确保同名仓库存在。如果 `create_missing_repos=false` 且目标不存在，跳过该仓库。创建时设置 `auto_init=false` 避免与 mirror push 冲突 | 创建失败 → 跳过该仓库 |
| **B. 同步 Git 代码** | 核心步骤，使用 `git clone --mirror` + `git push --mirror`。clone 后先通过 `git show-ref` 与目标端 `git ls-remote` 比较 refs，两端已一致则跳过推送（输出 "Both sides already in sync ⏭️"） | clone/push 失败 → 记录错误，跳过该仓库 |
| **C. 同步仓库元信息** | 同步描述、主页等基本信息。仅在有差异时调用 PATCH API | 非致命错误，记录日志继续 |
| **D. 同步 Wiki** | 使用 `git clone --mirror` + `git push --all/--tags --force` 同步 `.wiki.git` 仓库。clone 后比较两端 wiki refs，已一致则跳过推送（输出 "Wiki already in sync ⏭️"）。源仓库没有 Wiki 时静默跳过 | 非致命错误，记录日志继续 |
| **E. 同步 Releases** | 以 `tag_name` 为匹配键，同步发布信息和附件。附件需下载后重新上传 | 非致命错误，记录日志继续 |
| **F. 同步 Labels** | 以 `name` 为匹配键，同步标签名称、颜色、描述 | 非致命错误，记录日志继续 |
| **G. 同步 Milestones** | 以 `title` 为匹配键，同步里程碑标题、状态、描述、截止日期 | 非致命错误，记录日志继续 |
| **H. 同步 Issues** | 最复杂的同步项。仅同步 open 状态的 issues，通过标识匹配避免重复创建 | 非致命错误，记录日志继续 |

### 错误处理原则

1. **Git 代码同步失败 → 整个仓库标记为失败**（核心步骤不可跳过）
2. **附属信息同步失败 → 仅记录警告，不影响仓库整体同步结果**（增量功能，允许部分失败）
3. **所有步骤出错后继续处理下一个仓库**（不因单个仓库失败中断全局）
