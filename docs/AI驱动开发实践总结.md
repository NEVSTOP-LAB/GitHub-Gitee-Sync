# 用 AI 写完一整个项目，我学到了什么

[GitHub-Gitee-Sync](https://github.com/NEVSTOP-LAB/GitHub-Gitee-Sync) 这个项目，从第一行代码到能跑的 GitHub Action，全部由 AI 完成。我（人类）只负责提需求、看 PR、拍板。

花了 3 天，30 个 PR，最终产出约 6,600 行代码（含 3,200 行测试、181 个用例）。下面记录整个过程和踩过的坑。

---

## 项目背景

| 项 | 说明 |
|------|------|
| **做什么** | 在 GitHub 和 Gitee 之间同步所有仓库（代码、Wiki、Release、Label、Milestone 等） |
| **交付物** | Python 工具 + Docker 镜像 + GitHub Action |
| **代码量** | ~6,600 行（含 ~3,200 行测试），181 个测试用例 |
| **周期** | 3 天（2026-03-27 ~ 2026-03-30），30 个 PR |
| **AI** | GitHub Copilot（主力编码）、Claude（评审/修复）、Codex（评审/修复） |
| **人类** | 需求、方向、最终决策（NEVSTOP） |

---

## 整体流程

开发分成五步，像流水线一样往前推：

```
规划调研 ──→ 编码实现 ──→ 多方评审 ──→ 修复问题 ──→ 上线调优
 (PR #1)     (PR #2)    (PR #3~6)   (PR #7~12)  (PR #14~30)
```

| 阶段 | 时间 | 做了什么 | 产出 |
|------|------|---------|------|
| 规划调研 | 第 1 天晚上 | AI 调研 API、画流程图、写设计文档 | 12 篇文档，3,400+ 行 |
| 编码实现 | 第 2 天上午 | AI 按设计文档写完全部代码 | 4,800+ 行变更，144 个测试 |
| 多方评审 | 第 2 天中午 | 3 个 AI 各自评审 → 1 个 AI 汇总筛选 | 4 份评审报告 + 1 份汇总 |
| 修复问题 | 第 2 天下午~晚 | 根据评审报告修安全漏洞、逻辑错误 | 安全加固、功能增强 |
| 上线调优 | 第 2 天晚~第 3 天 | 实际跑起来后修各种 Bug | 15+ 个修复/增强 PR |

## 人和 AI 怎么配合

分工很简单：

- **我**：写 Issue 描述需求、看 PR 决定合不合、AI 搞错了就 Revert、偶尔改改 README 格式
- **AI**：调研、设计、写代码、写测试、互相评审、修 Bug

日常交互四种模式：

| 模式 | 场景 | 例子 |
|------|------|------|
| **Issue 驱动** | 提新需求或报 Bug | 我建 Issue → Copilot 自动开 PR 解决 |
| **Review 驱动** | 对 PR 提修改意见 | 我在 PR 里留评论 → AI 根据反馈改代码 |
| **日志驱动** | 实际运行报错 | 我把错误日志贴到 Issue → AI 定位修复 |
| **Revert 重做** | AI 方案根本跑偏 | 我 Revert 错误 PR → 重新描述需求让 AI 再来 |

## 各阶段详细说明

### 规划阶段：一句话变十二篇文档

我只写了一行 README，AI 自己完成了 API 调研、流程图、模块设计、错误处理方案、开发步骤 Checklist。

产出的文档清单：

| 类别 | 文档 | 内容 |
|------|------|------|
| 调研 | GitHub-API.md | REST API 分页、认证、仓库 CRUD |
| 调研 | Gitee-API.md | REST API v5 接口 |
| 调研 | GitHub-Actions.md | Docker Container Action 机制 |
| 调研 | Git-Mirror-同步机制.md | git clone --mirror + push 原理 |
| 调研 | 仓库附属信息同步调研.md | Release/Wiki/Label 等 API 可行性 |
| 设计 | Python-脚本设计.md | 模块/函数/参数设计 |
| 设计 | Docker-镜像设计.md | Dockerfile 设计 |
| 设计 | GitHub-Action-设计.md | action.yml 设计 |
| 设计 | 流程图.md | 整体流程 + 单仓库同步流程 |
| 设计 | 错误处理设计.md | 五大类错误的处理策略 |
| 设计 | 开发步骤.md | 分阶段开发计划（含 Checklist） |

这步很值——后面写代码时 AI 严格按文档来，方向基本没跑偏。

### 编码阶段：一个 PR 搞定 4,800 行

AI 严格按规划文档实现，一次交付所有代码：

| 产出 | 详情 |
|------|------|
| 模块化代码 | `sync.py` + `lib/`（4 个模块） |
| Docker 支持 | `Dockerfile` + `.dockerignore` |
| GitHub Action | `action.yml` + `entrypoint.sh` |
| CI 配置 | `.github/workflows/test.yml` |
| 测试 | 144 个单元测试（5 个测试文件） |

一次性大量生成的代码，后面确实需要多轮评审和修复。

### 评审阶段：三个 AI 交叉看代码

把同一份代码分别交给 Codex、Claude、Copilot 评审，三份报告各有侧重。然后再让 Copilot 做"二级评审"——合并、去重、过滤：

```
           ┌──── Codex ──── 评审报告（偏工程实践）
           │
代码 ──────┼──── Claude ──── 评审报告（偏架构安全）
           │
           └──── Copilot ──── 评审报告（偏安全正确性）
                     │
                     ▼
            Copilot 二级评审：汇总 + 过滤 + 优先级排序
```

过滤原则很关键：
- ❌ 丢掉重构建议（代码能跑，别折腾策略模式 / dataclass）
- ❌ 丢掉加复杂度的建议（并发、结构化日志、健康检查端点）
- ❌ 丢掉不合理的（API 没这能力、CLI 工具不需要 HTTP 端点）
- ❌ 去重（保留描述最详细的版本）
- ✅ 只留安全问题、正确性问题、可用性问题

评审发现的关键问题：
1. 🔴 Gitee Token 在 URL 查询参数中暴露
2. 🔴 双向同步存在数据覆盖风险
3. 🔴 Token 校验没复用统一请求封装
4. 🟡 Release Asset 只按名称去重，不校验内容
5. 🟡 日志中 Token 可能遗漏脱敏

### 修复阶段：按评审报告逐项修复

| PR | 主题 | 改动 |
|----|------|------|
| #7 | 安全+正确性（11 个修复） | Gitee 切 Bearer 认证、Token 校验复用、Rate Limit 防御性解析等 |
| #8 | 破坏性推送修复 | `git push --mirror` → 增量推送，不再删目标端独有内容 |
| #9 | 文档补充 | org scope 提示、非对称账号类型说明 |
| #10 | 功能增强 | sync-log 输出、include-repos 白名单 |
| #11 | 安全加固 | heredoc 注入修复、Token 全链路脱敏、响应体清洗 |
| #12 | PR 评论修复 | mask_token 分隔符、临时文件安全 |

### 上线阶段：评审发现不了的问题

这是**最有价值的阶段**。代码真正在 GitHub Action 里跑起来后，暴露出一堆评审阶段根本不可能发现的问题：

| 问题类别 | PR | 问题描述 |
|---------|-----|---------|
| 环境差异 | #15, #17 | Docker 里 `dash` shell 丢弃带连字符的环境变量（`INPUT_GITHUB-OWNER` 直接没了） |
| API 行为差异 | #19→#20→#21 | Gitee HTTPS 认证需要 token 创建者的个人 username，不是 org 名 |
| API 隐式要求 | #22 | Gitee PATCH API 请求体必须包含 `name` 字段（文档没写） |
| API 行为差异 | #27 | Gitee Release 创建需要 `target_commitish` 字段 |
| API 行为差异 | #28 | Gitee 分页超过最后一页返回重复数据（GitHub 返回空） |
| API 行为差异 | #29 | Gitee Label 颜色需要 `#` 前缀，GitHub 不需要 |
| 逻辑缺陷 | #30 | Release 要按创建时间正序排列再同步，不然最旧的变成"最新" |
| 性能优化 | #23 | 两端已同步时跳过冗余推送 |
| 隐私保护 | #24, #25 | 私有仓库名称在日志中脱敏 |

### 一次典型的 Revert 纠错（PR #19→#20→#21）

1. PR #19：AI 修 Gitee 403，方案是拿 owner 当 GIT_ASKPASS 的 username
2. PR #20：我 Revert — 因为 owner 是组织名，不是 token 创建者的个人账号，Gitee 不认
3. PR #21：我说清楚需求后，AI 改成通过 API 查 token 创建者的真实 login，问题解决

教训：AI 不理解的业务逻辑，靠反复评论纠正效率低。Revert + 说清楚需求，比改三轮快。

## 经验总结

### 先规划，别急着写代码

让 AI 先调研、出设计文档，看完确认方向再写代码。本项目的 3,400+ 行规划文档为后面的编码提供了清晰蓝图。跳过这步，后面返工代价更大。

### 多个 AI 交叉评审

一个 AI 有盲点。三个 AI 独立评审 + 一个 AI 汇总筛选，效果好很多。不同 AI 关注点不同：有的偏安全，有的偏工程实践，有的偏架构。

关键是二级评审的过滤原则——忽略过度设计的建议，只聚焦安全和正确性。

### 尽早跑起来

再怎么评审，也发现不了 Gitee API 那些没写进文档的"潜规则"（`name` 必填、颜色要 `#`、分页翻过头返回重复数据）。早跑早发现，把错误日志贴给 AI 是最快的修复方式。

### 该 Revert 就 Revert

AI 方向搞错了，别来回评论纠正。直接 Revert，把需求说清楚让它重做。PR #19→#20→#21 就是例子。Revert 还能保持 git 历史的清晰。

### 人管方向，AI 管实现

我不写代码，但关键决策我来拍：用增量推送还是 mirror？Token 怎么传？单仓库失败要不要阻塞全局？AI 负责把这些决策变成代码和测试。

### 一个 PR 一件事

小的、聚焦的 PR 好审查、好 Revert。PR #7 一次修了 11 个问题，虽然可以接受，但拆开会更好。

## AI 的局限性

| 局限性 | 表现 | 应对 |
|-------|------|------|
| 不了解平台"潜规则" | Gitee 的一堆未文档化要求 | 实际跑 + 贴日志给 AI |
| 可能只改表面 | 拿 org 名当 username（PR #19） | 人审查业务逻辑正确性 |
| 爱"过度设计" | 评审动不动建议加策略模式 | 二级评审过滤 |
| 一次性输出有遗漏 | 首版缺边界处理 | 多轮评审 + 实跑补全 |

## 推荐工作流

```
  提需求（一句话就行）
       │
       ▼
  AI 调研 + 出设计文档 ← 人审核确认方向
       │                   ★ 不要跳过这步直接让 AI 写代码
       ▼
  AI 写代码 + 测试 ← 要求 AI 同步写测试用例
       │
       ▼
  多个 AI 评审 → 汇总筛选 ← 过滤过度设计，聚焦安全/正确性
       │
       ▼
  AI 按评审结果改代码 ← 人审 PR，不对就 Revert
       │
       ▼
  实际跑 → 贴日志 → AI 修 → 再跑（循环直到稳定）
                              ★ 这是发现平台差异的唯一途径
```

核心原则：

| 原则 | 说明 |
|------|------|
| 人管 What，AI 管 How | 人定义"做什么"，AI 决定"怎么做" |
| 先规划后编码 | 让 AI 先输出调研/设计文档，审核后再编码 |
| 多重评审 | 多个 AI 交叉评审，二级评审过滤噪音 |
| 尽早实战 | 代码评审无法替代实际运行 |
| 果断 Revert | AI 方向错了就 Revert，别反复纠正 |
| 小步迭代 | 每个 PR 聚焦一个问题，便于审查和回滚 |
| 充分上下文 | 给 AI 提供错误日志、API 文档、期望行为 |

## 什么项目适合这么干

| 适合 ✅ | 不太适合 ⚠️ |
|---------|------------|
| 功能明确、边界清晰的工具类项目 | 需要深度领域知识的业务系统 |
| 有公开 API 文档的集成项目 | 复杂并发/分布式系统 |
| 标准技术栈（Python/Node/Go 等） | 小众框架或私有库 |
| 有测试基础设施的项目 | 对性能要求极致的底层系统 |
| 需要快速原型验证的项目 | 需要长期维护演进的核心系统 |
