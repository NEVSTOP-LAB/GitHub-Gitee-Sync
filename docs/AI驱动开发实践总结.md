# 用 AI 写完一整个项目，我学到了什么

[GitHub-Gitee-Sync](https://github.com/NEVSTOP-LAB/GitHub-Gitee-Sync) 这个项目，从第一行代码到能跑的 GitHub Action，全部由 AI 完成。我（人类）只负责提需求、看 PR、拍板。

花了 3 天，30 个 PR，最终产出约 6,600 行代码（含 3,200 行测试、181 个用例）。下面记录整个过程和踩过的坑。

---

## 项目背景

一句话：在 GitHub 和 Gitee 之间同步所有仓库（代码、Wiki、Release、Label 等），打包成 Python 工具 + Docker 镜像 + GitHub Action。

参与的 AI：GitHub Copilot（主力）、Claude、Codex。

---

## 整体流程

开发分成五步，像流水线一样往前推：

```
规划调研 ──→ 编码实现 ──→ 多方评审 ──→ 修复问题 ──→ 上线调优
 (PR #1)     (PR #2)    (PR #3~6)   (PR #7~12)  (PR #14~30)
```

| 阶段 | 时间 | 做了什么 |
|------|------|---------|
| 规划调研 | 第 1 天晚上 | AI 调研 API，画流程图，写设计文档（12 篇，3,400+ 行） |
| 编码实现 | 第 2 天上午 | AI 按设计文档写完全部代码（4,800+ 行变更，144 个测试） |
| 多方评审 | 第 2 天中午 | 3 个 AI 各自评审，再由 1 个 AI 汇总筛选 |
| 修复问题 | 第 2 天下午~晚 | 根据评审报告修安全漏洞、逻辑错误 |
| 上线调优 | 第 2 天晚~第 3 天 | 实际跑起来后的各种 Bug 修复（15+ 个 PR） |

---

## 人和 AI 怎么配合

分工很简单：

- **我**：写 Issue 描述需求、看 PR 决定合不合、AI 搞错了就 Revert、偶尔改改 README 格式
- **AI**：调研、设计、写代码、写测试、互相评审、修 Bug

日常交互就四种模式：

1. 我建 Issue → AI 开 PR 实现
2. 我在 PR 里留评论 → AI 改代码
3. 跑出错了，我把日志贴给 AI → AI 修
4. AI 改错了 → 我 Revert → 重新描述需求让 AI 再来

## 几个值得说的阶段

### 规划阶段：一句话变成十二篇文档

我只写了一行 README，AI 自己做了 API 调研（GitHub/Gitee 两端）、画了流程图、设计了模块结构和错误处理方案、排了开发步骤的 Checklist。

这步很值。后面写代码的时候 AI 严格按文档来，方向基本没跑偏。

### 评审阶段：三个 AI 交叉看代码

把同一份代码分别交给 Codex、Claude、Copilot 评审，三份报告各有侧重。然后再让 Copilot 做"二级评审"——把三份报告合并、去重、过滤掉不靠谱的建议。

过滤原则很关键：
- 丢掉重构建议（代码能跑，别折腾策略模式 / dataclass）
- 丢掉加复杂度的建议（并发、结构化日志、健康检查端点）
- 丢掉不合理的（API 没这能力、CLI 工具不需要 HTTP 端点）
- 只留安全问题、正确性问题、可用性问题

### 上线阶段：评审发现不了的问题

评审能挑出 Token 泄露、注入风险这类问题，但有一类问题只有真正跑起来才会碰到——平台 API 的"潜规则"：

| 问题 | 说明 |
|------|------|
| Gitee PATCH 必须带 `name` 字段 | 文档没写，不带就 400 |
| Gitee 颜色要 `#` 前缀 | GitHub 用 `008672`，Gitee 要 `#008672` |
| Gitee 分页翻过头返回重复数据 | GitHub 返回空，Gitee 返回上一页 |
| Gitee Release 要 `target_commitish` | GitHub 可选，Gitee 必填 |
| Docker 里 dash 丢弃带连字符的环境变量 | `INPUT_GITHUB-OWNER` 直接没了 |

### 一次典型的 Revert 纠错

PR #19：AI 修 Gitee 403，方案是拿 owner 当 GIT_ASKPASS 的 username。

问题：owner 是组织名，不是 token 创建者的个人账号，Gitee 不认。

我直接 Revert（PR #20），重新描述清楚："username 要用 token 创建者的 login，不是 org 名"。AI 第二次（PR #21）通过 API 查到正确的 login，问题解决。

教训：AI 不理解的业务逻辑，靠反复评论纠正效率低。Revert + 说清楚需求，比改三轮快。

## 经验总结

### 先规划，别急着写代码

让 AI 先调研、出设计文档，看完确认方向再写代码。跳过这步，后面返工的代价更大。

### 多个 AI 交叉评审

一个 AI 可能有盲点。三个 AI 独立评审 + 一个 AI 做汇总筛选，效果比单次评审好很多。记得过滤掉那些"正确但没必要"的重构建议。

### 尽早跑起来

再怎么评审，也发现不了 Gitee API 那些没写进文档的"潜规则"。早跑早发现，把错误日志贴给 AI 是最快的修复方式。

### 该 Revert 就 Revert

AI 方向搞错了，别来回评论纠正。直接 Revert，把需求说清楚让它重做。PR #19→#20→#21 就是例子。

### 人管方向，AI 管实现

我不写代码，但关键决策我来拍：用增量推送还是 mirror？Token 怎么传？单仓库失败要不要阻塞全局？AI 负责把这些决策变成代码。

### 一个 PR 一件事

小的、聚焦的 PR 好审查、好 Revert。PR #7 一次修了 11 个问题，虽然也能接受，但拆开会更好。

## AI 干不好的地方

| 弱项 | 例子 | 怎么办 |
|------|------|--------|
| 不懂平台"潜规则" | Gitee 的一堆未文档化要求 | 实际跑 + 贴日志给 AI |
| 可能只改表面 | 拿 org 名当 username | 人审查业务逻辑 |
| 爱"过度设计" | 评审动不动建议加策略模式 | 二级评审过滤 |
| 一次性输出有遗漏 | 首版缺边界处理 | 多轮评审 + 实跑补全 |

## 推荐工作流

```
  提需求（一句话就行）
       │
       ▼
  AI 调研 + 出设计文档 ← 人审核确认方向
       │
       ▼
  AI 写代码 + 测试
       │
       ▼
  多个 AI 评审 → 汇总筛选 ← 过滤掉过度设计
       │
       ▼
  AI 按评审结果改代码 ← 人审 PR，不对就 Revert
       │
       ▼
  实际跑 → 贴日志 → AI 修 → 再跑（循环直到稳定）
```

一句话版本：**人定方向，AI 干活，多轮评审，跑起来验证。**

---

## 什么项目适合这么干

合适的：边界清楚的工具类项目、有公开 API 的集成项目、标准技术栈、需要快速出原型的项目。

不太合适的：需要深度领域知识的业务系统、复杂并发/分布式系统、小众框架/私有库项目。
