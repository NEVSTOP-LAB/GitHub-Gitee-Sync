# 仓库附属信息同步调研

## 概述

除了 Git 仓库代码（分支、标签、提交历史）以外，GitHub/Gitee 仓库还包含大量附属信息。本文档调研这些附属信息在两个平台之间的可同步性，以及对应的 API 支持情况。

---

## 1. 可同步信息总览

| 信息类型 | GitHub API | Gitee API | 同步可行性 | 同步方式 | 优先级 |
|---------|-----------|----------|-----------|---------|-------|
| **仓库元信息**（描述、主页） | ✅ PATCH `/repos/{owner}/{repo}` | ✅ PATCH `/repos/{owner}/{repo}` | ✅ 可行 | REST API | P0 |
| **Releases（发布）** | ✅ 完整 CRUD | ✅ 完整 CRUD | ✅ 可行 | REST API | P1 |
| **Release Assets（发布附件）** | ✅ 上传/下载 | ✅ 上传/下载 | ✅ 可行（需下载后重新上传） | REST API + 文件传输 | P1 |
| **Wiki** | ❌ 无 REST API | ✅ 有 REST API | ⚠️ 部分可行 | Git clone `.wiki.git` | P2 |
| **Issues（议题）** | ✅ 完整 CRUD | ✅ 完整 CRUD | ⚠️ 可行但复杂 | REST API | P3 |
| **Issue Comments（评论）** | ✅ 完整 CRUD | ✅ 完整 CRUD | ⚠️ 可行但复杂 | REST API | P3 |
| **Labels（标签）** | ✅ 完整 CRUD | ✅ 完整 CRUD | ✅ 可行 | REST API | P2 |
| **Milestones（里程碑）** | ✅ 完整 CRUD | ✅ 完整 CRUD | ✅ 可行 | REST API | P2 |
| **Pull Requests** | ✅ 只读+创建 | ✅ 只读+创建 | ❌ 不建议同步 | - | - |
| **GitHub Actions / CI** | N/A | N/A | ❌ 平台绑定 | - | - |
| **GitHub Pages** | N/A | N/A | ❌ 平台绑定 | - | - |
| **Webhooks** | ✅ CRUD | ✅ CRUD | ❌ 不建议同步（配置不同） | - | - |
| **Deploy Keys** | ✅ CRUD | ✅ CRUD | ❌ 不建议同步（安全性） | - | - |

## 2. 仓库元信息

### 可同步字段

| 字段 | GitHub 字段名 | Gitee 字段名 | 说明 |
|------|-------------|-------------|------|
| 仓库描述 | `description` | `description` | 完全对应 |
| 主页地址 | `homepage` | `homepage` | 完全对应 |
| 默认分支 | `default_branch` | `default_branch` | 通过 API 更新 |
| 私有状态 | `private` | `private` | 完全对应 |

### API 端点

```
# GitHub：更新仓库
PATCH https://api.github.com/repos/{owner}/{repo}
Body: { "description": "...", "homepage": "..." }

# Gitee：更新仓库
PATCH https://gitee.com/api/v5/repos/{owner}/{repo}
Body: { "description": "...", "homepage": "..." }
```

### 同步策略

- 从源平台获取仓库详情，提取 `description`、`homepage` 等字段
- 在目标平台调用 PATCH API 更新仓库信息
- 仅在字段值不同时才调用更新 API，避免不必要请求

## 3. Releases（发布）

### 3.1 GitHub Releases API

```
GET    /repos/{owner}/{repo}/releases            # 列出所有 release
POST   /repos/{owner}/{repo}/releases            # 创建 release
PATCH  /repos/{owner}/{repo}/releases/{id}       # 更新 release
DELETE /repos/{owner}/{repo}/releases/{id}       # 删除 release
```

**Release 关键字段**：
- `tag_name` — 关联的 Git 标签
- `name` — 发布标题
- `body` — 发布说明（Markdown）
- `draft` — 是否为草稿
- `prerelease` — 是否为预发布
- `assets` — 附件列表

### 3.2 Gitee Releases API

```
GET    /repos/{owner}/{repo}/releases            # 列出所有 release
POST   /repos/{owner}/{repo}/releases            # 创建 release
PATCH  /repos/{owner}/{repo}/releases/{id}       # 更新 release
DELETE /repos/{owner}/{repo}/releases/{id}       # 删除 release
```

**请求体**（创建/更新）：
```json
{
  "access_token": "...",
  "tag_name": "v1.0.0",
  "name": "Release Title",
  "body": "Release notes...",
  "prerelease": false
}
```

### 3.3 Release Assets（附件）同步

**GitHub**：
```
GET  /repos/{owner}/{repo}/releases/{id}/assets           # 列出附件
POST https://uploads.github.com/repos/{owner}/{repo}/releases/{id}/assets?name=xxx  # 上传附件
```

**Gitee**：
```
POST /repos/{owner}/{repo}/releases/{id}/attach_files     # 上传附件（multipart/form-data）
```

### 3.4 Release 同步策略

1. 从源平台获取所有 releases（按 `tag_name` 索引）
2. 从目标平台获取已有 releases
3. 对比差异：
   - 源有目标无 → 在目标平台创建
   - 两端都有 → 比较 `name`、`body`、`prerelease` 字段，有差异则更新
   - 源无目标有 → 根据配置决定是否删除（默认不删除，避免误删）
4. 同步 assets：
   - 下载源平台附件到临时文件
   - 上传到目标平台
   - 按文件名匹配，跳过已存在的同名附件

## 4. Wiki

### 4.1 GitHub Wiki

GitHub **没有** Wiki REST API。Wiki 本质是一个独立的 Git 仓库：

```
https://github.com/{owner}/{repo}.wiki.git
```

**同步方式**：使用 `git clone --mirror` + `git push --mirror`，与代码仓库同步方式一致。

```bash
# 克隆 GitHub Wiki
git clone --mirror https://<token>@github.com/{owner}/{repo}.wiki.git /tmp/{repo}.wiki.git

# 推送到 Gitee Wiki（需要先在 Gitee 上手动或通过 API 初始化 Wiki）
git push --mirror https://<token>@gitee.com/{owner}/{repo}.wiki.git
```

### 4.2 Gitee Wiki

Gitee **有** Wiki REST API：

```
GET    /repos/{owner}/{repo}/wikis            # 列出 wiki 页面
GET    /repos/{owner}/{repo}/wikis/{sha}      # 获取特定页面
POST   /repos/{owner}/{repo}/wikis            # 创建页面
PATCH  /repos/{owner}/{repo}/wikis/{sha}      # 更新页面
```

Gitee Wiki 也是 Git 仓库，同样支持 `git clone` 方式访问。

### 4.3 Wiki 同步策略

- **推荐方案**：使用 `git clone --mirror` + `git push --mirror`（与代码仓库同步一致）
- **前提条件**：目标仓库需要已启用 Wiki（至少有一个页面或已手动初始化）
- **注意事项**：
  - 如果源仓库没有 Wiki（`.wiki.git` 不存在），clone 会失败，需捕获错误并跳过
  - 空 Wiki 仓库无法 push

## 5. Issues（议题）

### 5.1 API 端点

**GitHub**：
```
GET    /repos/{owner}/{repo}/issues            # 列出 issues
POST   /repos/{owner}/{repo}/issues            # 创建 issue
PATCH  /repos/{owner}/{repo}/issues/{number}   # 更新 issue
GET    /repos/{owner}/{repo}/issues/{number}/comments  # 列出评论
POST   /repos/{owner}/{repo}/issues/{number}/comments  # 创建评论
```

**Gitee**：
```
GET    /repos/{owner}/{repo}/issues            # 列出 issues
POST   /repos/{owner}/{repo}/issues            # 创建 issue
PATCH  /repos/{owner}/{repo}/issues/{number}   # 更新 issue
GET    /repos/{owner}/{repo}/issues/{number}/comments  # 列出评论
POST   /repos/{owner}/{repo}/issues/{number}/comments  # 创建评论
```

### 5.2 Issues 同步难点

| 问题 | 说明 |
|------|------|
| Issue 编号不可控 | 目标平台的 issue number 由系统自动分配，无法指定 |
| 作者信息丢失 | 在目标平台创建的 issue 只能以 Token 所有者身份创建 |
| 关联信息丢失 | 跨 issue 引用（#123）在目标平台编号不同 |
| Assignees 不同 | 两个平台的用户系统不同，无法映射 |
| 状态同步 | open/closed 状态需要单独处理 |
| 幂等性 | 重复运行时需要避免重复创建 |

### 5.3 Issues 同步策略

由于复杂度高，建议作为 **可选功能**，且默认不启用：

1. 以 `tag_name` 或标题 + 创建时间作为匹配键
2. 在 issue body 末尾附加源链接作为标识
3. 同步时检查是否已同步过（通过标识判断）
4. 仅同步 open 状态的 issues，避免大量历史数据

## 6. Labels（标签）

### API 端点（GitHub 和 Gitee 一致）

```
GET    /repos/{owner}/{repo}/labels            # 列出所有标签
POST   /repos/{owner}/{repo}/labels            # 创建标签
PATCH  /repos/{owner}/{repo}/labels/{name}     # 更新标签
DELETE /repos/{owner}/{repo}/labels/{name}      # 删除标签
```

### 同步策略

- 以 `name` 作为匹配键
- 同步 `name`、`color`、`description` 字段
- 源有目标无 → 创建
- 两端都有 → 比较 `color` 和 `description`，有差异则更新
- 源无目标有 → 默认保留（避免误删用户自定义标签）

## 7. Milestones（里程碑）

### API 端点（GitHub 和 Gitee 一致）

```
GET    /repos/{owner}/{repo}/milestones            # 列出里程碑
POST   /repos/{owner}/{repo}/milestones            # 创建里程碑
PATCH  /repos/{owner}/{repo}/milestones/{number}   # 更新里程碑
DELETE /repos/{owner}/{repo}/milestones/{number}    # 删除里程碑
```

### 关键字段

- `title` — 里程碑标题
- `state` — `open` / `closed`
- `description` — 描述
- `due_on` — 截止日期

### 同步策略

- 以 `title` 作为匹配键
- 同步 `title`、`state`、`description`、`due_on` 字段
- 源有目标无 → 创建
- 两端都有 → 比较各字段，有差异则更新

## 8. 同步优先级与实施建议

### 实施阶段

| 阶段 | 同步内容 | 优先级 | 复杂度 |
|------|---------|-------|-------|
| **第一阶段（核心）** | Git 仓库代码（分支、标签） | P0 | 低 |
| **第一阶段（核心）** | 仓库元信息（描述、主页） | P0 | 低 |
| **第二阶段（增强）** | Releases + Assets | P1 | 中 |
| **第二阶段（增强）** | Wiki（Git mirror 方式） | P2 | 低 |
| **第三阶段（扩展）** | Labels | P2 | 低 |
| **第三阶段（扩展）** | Milestones | P2 | 低 |
| **第四阶段（可选）** | Issues + Comments | P3 | 高 |

### 配置项设计

建议通过参数控制同步范围：

```
--sync-extra "releases,wiki,labels,milestones,issues"
```

或环境变量：

```
SYNC_EXTRA=releases,wiki,labels,milestones,issues
```

默认值为空（仅同步代码和仓库元信息），用户按需启用。
